# Installation

Burp WAF Detector is currently distributed as source for manual installation.
It requires Burp Suite with Python-extension support and a configured Jython
standalone JAR.

## Requirements

- Burp Suite with the legacy Extender API available.
- A Jython standalone JAR configured as Burp's Python environment.
- A complete checkout of this repository. `BurpExtender.py`, `wafd/`, and
  `data/` must remain together because the extension loads both catalogues at
  runtime using paths relative to the package.

The extension itself has no third-party Python dependencies. The test suite
runs under CPython, while Burp loads the extension under Jython.

## Configure Jython

1. Obtain a Jython standalone JAR from a trusted source.
2. Open Burp's extension settings.
3. Under the Python environment settings, select the standalone JAR.
4. Apply the setting before loading the extension.

Burp's interface can change between releases. Refer to PortSwigger's
[extension settings documentation](https://portswigger.net/burp/documentation/desktop/settings/extensions)
if the Python environment controls are not in the expected location.

## Load the extension

1. Open **Extensions > Installed**.
2. Select **Add**.
3. Choose **Python** as the extension type.
4. Select the repository's `BurpExtender.py` file.
5. Load the extension and inspect its **Output** and **Errors** tabs.

A successful load prints a message similar to:

```text
Burp WAF Detector loaded with 38 rules
```

The **WAF Detector** suite tab and **Probe for WAF** request context-menu item
should then be available. PortSwigger also documents the current
[manual extension installation process](https://portswigger.net/burp/documentation/desktop/extend-burp/extensions/installing/manual-install).

## Updating the catalogues

The bundled files are read when the extension registers:

- `data/default_rules.json`
- `data/probes.json`

After changing either file, unload and reload the extension. Invalid catalogue
JSON prevents that catalogue from loading; inspect Burp's extension error
output for the Python exception.

## Troubleshooting

### The extension does not load

- Confirm that Jython is configured in Burp.
- Confirm that `BurpExtender.py` was selected as a Python extension.
- Keep the `wafd` and `data` directories beside the entry-point file.
- Validate the JSON catalogues with:

  ```bash
  python3 -m json.tool data/default_rules.json >/dev/null
  python3 -m json.tool data/probes.json >/dev/null
  ```

### The tab appears but no issues are created

- Confirm **Passive monitoring > Enabled** in the extension tab.
- By default, passive processing is restricted to URLs in Burp's target scope.
- An origin receives its first current-assessment issue only after an eligible
  response has passed through the HTTP listener.
- Traffic generated by the extension itself is excluded from passive analysis.

### Active scanning sends fewer probes than expected

- The default maximum is three eligible catalogue entries per operation.
- Entries are selected in catalogue order.
- Each probe's `safe_methods` list must include the request method.
- Scanner insertion points whose names contain `cookie`, `authorization`, or
  `header` are deliberately skipped by the planner.

### Settings do not survive reload

The passive-enabled flag and confidence threshold are persisted using Burp's
extension settings. Rule checkboxes change the in-memory catalogue for the
current load only. To make rule enablement persistent, edit the rule's
`enabled` field in `data/default_rules.json` and reload the extension.
