# Features and behaviour

This document describes behaviour implemented by the current Burp adapter. It
distinguishes runtime functionality from descriptive metadata in the JSON
catalogues.

## Passive detection

The extension registers an HTTP listener and analyses responses whenever:

- passive monitoring is enabled;
- the message is a response rather than a request;
- the message was not generated by the extension itself; and
- the URL is in Burp scope when `in_scope_only` is enabled (the default).

The listener normalises the response into a bounded fingerprint and evaluates
every enabled rule that can operate without a control response. Passive
detection does not send additional requests.

Malformed messages and detector failures are reported to Burp's extension
error output rather than interrupting Burp's traffic processing.

## Active detection

Active detection runs only through one of these explicit routes:

- Burp Scanner calls the extension's active scanner check for an insertion
  point.
- The user chooses **Probe for WAF** for a selected request.

For Scanner operations, the planner selects enabled probes whose
`safe_methods` include the selected request method. It skips insertion-point
names containing `cookie`, `authorization`, or `header`. Selection follows
catalogue order and stops at `max_probes`, which defaults to three.

The context-menu route is intended to create a `wafd_probe` query parameter and
reuse the same active scanner routine while preserving the original request
body. Its current implementation appends to the complete request-line string,
which can place the marker after the HTTP version and produce a malformed
request. Treat the menu action as a known defect until target-aware request-line
construction and a Burp-adapter regression test are added. The route labels its
synthetic insertion point as `query`, so the Scanner-specific sensitive-name
check does not apply to it.

### Control and probe comparison

Active rules compare a probe response with a control response. For GET, HEAD,
and OPTIONS, the selected request/response is the control and Burp's insertion
point builds each probe request.

For other methods, the default `non_get_target` policy is `root`:

1. The extension retains the HTTP method and service.
2. It changes the request target to `/`.
3. It sends an empty-body root request as the control.
4. It sends the raw probe value as the root request body.

If the root control returns no response, the selected message becomes the
fallback control. Setting `non_get_target` to `selected` preserves the selected
path and delegates probe placement to Burp's insertion point.

The extension does not infer whether `/` is inert. It does not add an
application-specific safety header, negotiate an echo contract, or verify that
the application avoided state changes.

### Executed profile fields

The adapter currently executes only these optional `profile` fields:

| Field | Runtime effect |
| --- | --- |
| `accept` | Adds or replaces the request's `Accept` header. |
| `request_headers` | Adds or replaces the declared request headers. |

Other profile fields—including `content_type`, `content_types`, `endpoint`,
`insertion_points`, `body_size`, expected-response fields, classification,
notes, and safety descriptions—are catalogue metadata. They document intended
research coverage but do not currently construct specialised requests or
classify results.

In particular, a catalogue entry labelled GraphQL, multipart, cookie, header,
XML, or a specific content type is not proof that the adapter has built that
wire representation. The transmitted location still depends on root mode or
Burp's selected insertion point.

## Response fingerprints

The adapter records or derives:

| Field | Behaviour |
| --- | --- |
| Status | Integer HTTP status, or zero for a no-response observation. |
| Headers | Case-normalised response-header mapping. |
| Cookies | Names parsed from `Set-Cookie`; values are not retained. |
| Body length | UTF-8 byte length of the bounded body supplied to the fingerprint builder. |
| Body hash | SHA-256 of that bounded body. |
| Body sample | First 4,096 decoded characters. |
| HTTP version | Parsed from an `HTTP/x` response status line when available. |
| Connection state | `complete` normally; active no-response paths use `no-response`. |
| Elapsed time | Model field exists, but the Burp adapter does not currently populate it. |

The Burp adapter supplies at most the first 1 MiB of decoded body content to
the fingerprint builder. This bounds memory and matching work, but means the
recorded length and hash are not full-response values for larger bodies.

## Evidence and confidence

Rules create evidence containing a rule ID, origin, human-readable detail,
optional provider, source (`passive` or `active`), and optional action.

Evidence is deduplicated by rule ID per origin and capped at 100 observations.
The confidence engine then:

1. Finds the highest enabled weight available in every evidence group.
2. Uses their sum as the maximum possible score.
3. Counts only the strongest matched rule from each group.
4. Divides matched weight by maximum weight and caps the result at 100%.

The default WAF threshold is 60%. Provider scores use the same global
denominator and include only matched evidence tagged for that provider.

Headers and generic error statuses carry intentionally low standalone weights.
Behavioural transitions and explicit vendor action signatures carry more
weight. The exact bundled weights are described in
[Rules, probes, and confidence](rules-and-probes.md).

## Current-assessment issues

After each eligible response has been observed, the extension publishes a
Burp Scanner issue named **WAF Detector: current assessment** for that origin.
Burp is asked to replace the earlier issue with the new assessment, leaving one
current view rather than a growing history.

The issue contains:

- `WAF suspected` when confidence is at or above the configured threshold;
- `No WAF indicators detected` otherwise;
- confidence and threshold percentages;
- provider/edge signals ordered by provider score;
- observed security-action tags;
- distinct rule IDs and escaped evidence details; and
- a compact list of evidence IDs.

Burp confidence labels are mapped independently of the configurable threshold:

| Score | Burp confidence label |
| --- | --- |
| 85% or greater | Certain |
| 60% to below 85% | Firm |
| Below 60% | Tentative |

The representative request/response is the most recently stored message for
the origin. The assessment is held in memory and is not restored after the
extension unloads or Burp exits.

## Provider and action separation

The rule model deliberately keeps provider and security action as separate
tags. Examples of defensible conclusions are:

- Cloudflare edge marker observed; Cloudflare challenge confirmed only when
  `cf-mitigated: challenge` also matches.
- Akamai edge denial observed; App & API Protector is not proven as the cause.
- AWS WAF challenge or CAPTCHA confirmed when `x-amzn-waf-action` carries the
  documented action; a generic AWS-hosted 403 remains weak evidence.

The issue currently displays provider and action collections rather than a
multi-level prose conclusion. Analysts must still consider alternative edge,
bot-management, rate-limiting, ACL, origin, and proxy causes.
