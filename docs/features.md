# Features and behaviour

This document describes behaviour executed by the current Burp adapter. See
[Rules, probes, and confidence](rules-and-probes.md) for catalogue contents and
[Configuration](configuration.md) for every user setting.

## Passive detection

The HTTP listener analyses a response when:

- passive monitoring is enabled;
- the message is a response;
- the message was not generated by this extension; and
- the URL is in Burp scope when `in_scope_only` is enabled (the default).

Passive detection does not send requests. It normalises a bounded response
fingerprint, evaluates enabled rules that do not require a control, updates the
origin assessment, and publishes its current issue. An analysis failure is
reported to Burp's extension error output without interrupting Burp traffic.

## Active detection

Active requests are sent only when:

- Burp Scanner invokes the active scanner check; or
- the user chooses **Probe for WAF** for a selected message.

The planner evaluates every enabled concrete probe. For a legacy
insertion-point probe, the selected request method must appear in
`safe_methods`. A constructed profile may declare its own outgoing method; that
method must appear in its own allowlist regardless of the selected request's
method. There is no global non-idempotent-method override.

Scanner insertion points whose names contain `cookie`, `authorization`, or
`header` are skipped by the legacy insertion-point route. Specialist cookie
and header profiles remain available because they explicitly name the field
they are intended to modify.

### Context-menu requests

**Probe for WAF** reuses the active scanner routine. Legacy query probes are
placed in a dedicated `wafd_probe` parameter. The request line is parsed into
method, target, and HTTP version so the parameter cannot be appended after the
version token. Existing queries, absolute-form targets, fragments, HTTP/1.x,
and HTTP/2-form status tokens are covered by tests. CONNECT authority-form and
OPTIONS asterisk-form targets are rejected rather than malformed.

The original request body is preserved for legacy context-menu query probes.
Specialist profiles may deliberately construct another method, body, header,
cookie, or content type.

## Controls and target paths

Legacy GET/HEAD/OPTIONS probes use the selected request/response as their
shared control. Legacy non-GET probes use an empty-body request to `/` as the
control when `non_get_target` is `root`; if that control has no response, the
selected response is used as the fallback.

Every constructed specialist profile normally sends a same-shape control:

1. The method, target policy, placement, content type, field name, multipart
   form, and declared headers match the probe.
2. The control contains `ordinary` or the profile's explicit `control_value`.
3. The probe changes the logical value, or a declared size/boundary offset.
4. Repeated transmissions reuse the first control for that concrete probe ID.

Constructed non-GET profiles target `/` by default, including when active
probing was triggered from a GET request. `non_get_target=selected` preserves
the selected target. The UI labels these choices **Root path (/)** and
**Selected request path**, respectively. An explicit profile endpoint takes
precedence.

## Specialist request construction

The request builder currently executes these placements:

| Placement | Construction |
| --- | --- |
| Query | Replaces the first named parameter or appends it if absent. |
| Query pairs | Preserves declared duplicate names and ordering. |
| Raw query | Appends bounded malformed/structural query syntax verbatim. |
| Form | URL-encoded key/value body. |
| JSON | Compact JSON object with a declared key. |
| GraphQL variables | Shallow query plus a JSON variables object. |
| GraphQL query | Shallow direct value in JSON or `application/graphql`. |
| XML | Escaped text inside a declared element. |
| SOAP | Escaped text inside a SOAP 1.2 envelope. |
| Header | Adds or replaces one explicitly named header; CR/LF is rejected. |
| Cookie | Adds a named encoded cookie while retaining existing cookies. |
| Repeated cookies | Sends repeated names in one header or multiple Cookie headers. |
| Multipart | Fields, filenames, file content, empty filenames, repeated fields, and multiple files. |
| Raw body | Sends the marker with a declared media type. |
| Sized body/header | Generates content around configured test thresholds. |
| Header count/total size | Generates bounded numbers or aggregate header sizes. |
| Inspection body | Places a harmless marker around the configured inspection boundary. |

`Content-Length` is removed from constructed headers so Burp recalculates it.
Client-supplied multipart filenames remain wire data and are never used as
local filesystem paths. XML text placements escape markup; the dedicated
internal-entity profile contains no external, file, or network entity.

Profile fields describing expected statuses, headers, body terms, safety, or
research notes remain metadata. Detection is performed by the shared rule
catalogue rather than executing expected-response expressions from a probe.

## Implemented matrices

The schema-version-2 catalogue expands compact values by placements for:

- five SQL-injection-shaped values across eight placements;
- five XSS-shaped values across eleven placements;
- nine body media types;
- GraphQL JSON and `application/graphql` direct-query forms;
- duplicate parameters in both orders;
- neutral `cmd` and `description` parameter-name comparisons;
- CRLF, null byte, invalid percent, empty value, repeated delimiters, and empty
  parameter names;
- GET, POST, HEAD, OPTIONS, PUT, DELETE, PATCH, TRACE, `WAFTEST`, and
  `X-HTTP-Method-Override: DELETE`;
- multipart parser variants;
- cookie/session variants;
- configured request and header-size boundaries;
- inspection marker positions; and
- XML entity, command, PHP, and absolute-path markers.

Queries use URL encoding unless a malformed-input profile explicitly requests
raw encoding. GraphQL queries are deliberately shallow and non-recursive.

## Response fingerprints

| Field | Behaviour |
| --- | --- |
| Status | Integer HTTP status, or zero for a transport failure. |
| Headers | Lower-case names; repeated values joined deterministically. |
| Cookies | Names plus SHA-256 value fingerprints; raw values are redacted. |
| Body length | UTF-8 byte length of the bounded body supplied to the builder. |
| Body hash | SHA-256 of that bounded body. |
| Body sample | First 4,096 decoded characters. |
| HTTP version | Parsed from the response status line when available. |
| Connection state | `complete`, `no-response`, `timeout`, `reset`, or `network-error`. |
| Elapsed time | Milliseconds measured around active callback requests. |

At most the first 1 MiB of decoded response body is supplied to the fingerprint
builder. Length and hash therefore describe that prefix for a larger response.

## Evidence and confidence

Evidence records rule ID, origin, detail, source, optional provider/action,
concrete probe characteristic, and optional classification. Passive evidence
is deduplicated by rule ID; active evidence is deduplicated by rule ID plus
concrete probe ID. Storage is bounded at 5,000 observations per origin.

A zero-weight `generic.probe-outcome` observation records each completed
control/probe status pair. It makes permitted or unchanged outcomes visible
without increasing WAF confidence. Malformed-input profiles are labelled
`malformed-request` separately from attack-shaped probes.

Confidence adds the strongest matched rule weight in each evidence group and
caps the result at 100%. Rule aliases therefore cannot stack, unrelated
provider catalogues cannot dilute an assessment, and audit observations with
weight zero do not contribute. The default WAF threshold is an inclusive 60%.

## Current-assessment issues

After passive observation or an active batch, the extension publishes
**WAF Detector: current assessment** for the origin. Burp replaces the previous
issue with the current view.

The issue contains:

- WAF suspected/no-indicators state;
- confidence and configured threshold;
- provider/edge signals;
- observed security actions;
- rule and concrete probe IDs;
- malformed-request classifications where applicable; and
- escaped response-derived details.

An active matrix publishes once after the batch rather than replacing the same
issue after every request. Burp confidence labels remain: Certain at 85% or
above, Firm at 60–84%, and Tentative below 60%.

## Provider and action separation

Provider and action tags remain independent. Examples:

- `cf-ray` is weak Cloudflare edge evidence; `cf-mitigated: challenge` confirms
  a Cloudflare challenge action.
- An Akamai reference page supports an Akamai edge denial but cannot prove App
  & API Protector rather than bot, rate, reputation, ACL, or property policy.
- `x-amzn-waf-action: challenge|captcha` is strong AWS WAF action evidence; a
  generic CloudFront 403 is not.

External wire evidence cannot replace vendor logs for products such as Google
Cloud Armor or generic ModSecurity/CRS deployments.
